## Architectural decision: lazy boundary in shared CanvasShell

The lazy split lives in `CanvasShell` which is consumed by ALL three embed modes. Tradeoffs:

| Mode | Lazy benefit | Cost |
|---|---|---|
| **L** Lit embed (`agentable-canvas/embed`) | None — `vite.embed.config.ts` has `inlineDynamicImports: true` (single-file delivery contract for the OSS embed bundle). Vite inlines the dynamic chunks. | +1 KB in main bundle (chunk-loader runtime). 248 KB ESM / 202 KB UMD, both well within budget (88-88%). |
| **B** React wrapper (`agentable-canvas/react`) | Real — host's bundler emits separate chunks. | None. |
| **C** Pure-React canvas (`agentable-canvas/react-canvas`) | Real — host's bundler emits separate chunks. The 7 secondary panels load on demand when user clicks NavSidebar icon. **Sandals website's production path.** | None. |

**Tradeoff accepted**: 1 KB cost in mode L for genuine benefit in modes B/C. The OSS embed bundle's "drop one tag, get one file" promise stays intact (no chunk-fetching boilerplate visible to consumers).

## What this unlocks for mode C consumers

When Sandals' website builds production, Vite emits the canvas as:
- Main chunk (eager): ~150-180 KB containing CanvasShell + ChatPanel + ArtifactsPanel + VoiceWidget + chrome
- 7 lazy chunks (one per panel): each ~10-30 KB, fetched on first NavSidebar click

User on first paint downloads only the main chunk. Settings (rarely opened), Applications (only after resume upload), Growth Paths (after specific conversation), etc. defer until needed.

This is the architect-flagged ~160-180 KB initial chunk target from the bundle-budget review.
